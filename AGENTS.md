# Shapez2 in Lean

## 応答言語
- 質問への回答・最終的な応答は日本語で行う
- 内部思考・サブエージェント内の応答や指示はトークン数削減のため英語可
- 日時・時刻を記載する場合は、実行中の PC のローカル時刻（UTC ではなく）を用いること
  - 日付: `YYYY-MM-DD` 形式（例: `2026-04-04`）
  - 時刻: `hh:mm:ss` 形式（例: `14:30:00`）
- ドキュメントの `最終更新` / `作成日` を記入・変更する際は、**必ず当日のローカル日付**を使用すること
  - 前セッションのサマリーや記憶に記録された日付をそのままコピーしない
  - 会話サマリーに埋め込まれた日付文字列は**サマリー生成時刻のもの**であり作業日と一致しない。信頼しないこと
  - 日付が不確かな場合は `run_in_terminal` で `Get-Date -Format 'yyyy-MM-dd'` を実行して確認すること
  - `作成日` より古い `最終更新` は論理的に不可能。コミット前に必ず確認すること

## 品質基準

本プロジェクトでは、以下の優先順位で品質を評価する。

1. **数学的美しさ**: 証明の構造・抽象度・一般性を最優先する。冗長な場合分けより、構造的な帰納法やジェネリックな補題を好む
2. **コード的な美しさ**: ファイル構成・命名・可読性。数学的美しさに影響しない範囲で改善する

リファクタリング判断時は、この順序を守ること。コードの見た目を整えるために証明の数学的構造を崩してはならない。

## エージェント動作ガイドライン

### 検索戦略

- **`semantic_search` はこのプロジェクトでは使用しない**。Lean コードの専門用語にエンベディングモデルが対応しておらず、空振り率が非常に高い。`grep_search` + `file_search` を優先すること
- コードベース探索時は、まず [`docs/s2il/s2il-lemma-index.md`](docs/s2il/s2il-lemma-index.md) の「ファイル構造マップ」と「依存関係マップ」を読み、対象の行範囲を特定してから `read_file` すること。SettleFoldlEq.lean は ~3,950 行あり、盲目的な部分読みは非効率
- **3,000 行超のファイルで複数箇所の行境界を調べる場合は、Explore サブエージェントに委譲すること**。メイン会話で `read_file` を 10 回以上繰り返す探索は非効率。サブエージェントに「対象の定理名リスト + 必要な情報（開始行・終端行・シグネチャ）」を指定し、結果をセッションメモリに記録する
- **`read_file` の最小幅: 50 行未満の読み取りは非推奨**。100 行単位を基準とし、コンテキスト把握のために前後を十分に含めること。4-5 行の読み取りを繰り返すよりも、一度に 200 行読む方がコンテキスト消費が少ない
- **`read_file` の重複省略**: 同一ファイルを 2 回以上読む場合、直前の読み取りと重複する行範囲は省略する。差分区間のみ読むこと
- Explore サブエージェントを使う際は、「既に判明している事実」を明記し、重複検索を防ぐこと

### Explore サブエージェントの適切な使い分け

Explore サブエージェントは**読み取り専用のコードベース探索**に特化している。以下の使い分けを厳守すること。

| 用途 | Explore で委譲 | REPL / ビルドで検証 |
|---|---|---|
| 定理の行番号・シグネチャ特定（3,000 行超 Lean ファイル） | ✅ | — |
| import チェーンの追跡 | ✅ | — |
| コードパターンの検索・集計 | ✅ | — |
| **`docs/` 配下のマークダウン（100–400 行程度）** | ❌ `read_file` で直接参照 | — |
| **行範囲が既知のファイルへのアクセス** | ❌ `read_file` で直接参照 | — |
| **simp 引数の必要性判定** | ❌ | ✅ |
| **証明タクティクの妥当性判定** | ❌ | ✅ |
| **型チェックが必要な判断** | ❌ | ✅ |

**原則**: Explore は**構文的・構造的な情報収集**に使い、**意味論的な判断**（型チェック・証明検証・simp 引数の有効性）には REPL またはビルドを使う。

**Explore 強制発動条件**: 以下の条件を**すべて**満たす場合、メイン会話での直接探索を中止し Explore サブエージェントに委譲すること:

1. 対象ファイルが 3,000 行超
2. 同一ファイルに対して `grep_search` または `read_file` を 3 回以上実行済み
3. まだ目的の情報が見つかっていない

**複数シンボルの位置特定（3 件以上）が必要なとき**は、ファイルサイズに関わらず Explore への委譲を推奨する。複数 grep → 複数 read の直列往復よりも Explore 1 往復の方がコンテキスト消費が少ない。

同一ファイルへの `grep_search` は **2 回/ファイル** を上限とする。3 回目が必要になったら Explore に委譲する。

検索スコープの拡大（例: ファイルスコープ → プロジェクト全体）は **1 回まで**。グローバル検索でシンボルが見つかったら即停止する。

**Explore 返答フォーマット制約**: 結果が見つかった項目のみを返すこと。「存在しない補題」「NOT FOUND 項目」のリストは含めない。

### セッションメモリの作成義務

> **最重要ルール: 多段作業（Wave・証明チェーン等）を開始したら、最初のツール呼び出し 3 回以内にセッションメモリを `/memories/session/` に作成すること。** PreCompact フックが発火した時点で未保存なら、即座に保存を実行する。

長時間の証明作業では、以下のタイミングで `/memories/session/` にメモを作成すること。
会話圧縮が発生するとコンテキストが失われるため、早めの保存が重要。

1 セッションにつき原則 **2 ファイル以内**。テンプレート・詳細ルールは [`docs/agent/session-memory-guide.md`](docs/agent/session-memory-guide.md) を参照。

**大型添付ドキュメントのトークン節約**: 5,000 tokens を超える計画書・仕様書を参照する作業では、セッション開始時に必要な箇所のみをセッションメモリに要約記録し、以降はそのメモを参照する。毎ターン全文を再読するのを避けること。

### 証明作業の撤退閾値

- **同一 sorry に対してアプローチが累計 8 回以上失敗した場合、`lean-proof-progress` スキルをロードして正式撤退判断を行うこと**。残り時間はドキュメント更新・次 sorry の分析・コミットに充てる
- **同一ゴール状態の再出現**: 戦略変更後に sorry のゴール形状が過去と同一に戻った場合、進捗なしと判定して即座に別戦略へ移行する。2 回再出現で撤退検討
- この閾値は「ビルドエラー修正」ではなく「証明戦略の試行」に対して適用する（simp 引数調整等の小修正はカウントしない）
- 撤退後は `docs/plans/gravity-proof-execution-plan.md` 等の実行計画に「試みたアプローチと失敗理由」を記録すること

### リポジトリメモリ (`/memories/repo/`) の更新方式

- 既存セクションの `str_replace` は長文マッチで失敗しやすい。**日付付きセクションの追記** (`insert`) を推奨する
  - 例: `## Wave 3 後の状態 (2026-04)` を末尾ではなく該当セクションの直後に追記
  - 旧セクションは履歴として残し、最新セクションを先頭に配置する

### タスク種別ごとのプラン提示粒度

実装開始前にプランを提示する場合、タスク種別に応じた粒度を守ること。過剰な詳細はユーザーの認知負荷を増やし、実態と合わない丁寧さはノイズになる。

| タスク種別 | 提示内容 | 不要なもの |
|---|---|---|
| ドキュメント変更のみ | 変更ファイルリスト + 追加・変更ルールの箇条書き | Phase 分割・詳細 Consideration セクション |
| コード変更（小〜中） | 変更ファイルリスト + 変更の要点 | 証明戦略レベルの詳細分析 |
| 証明作業（Wave 等） | 現状の sorry 数・Wave の目標・対象定理リスト | — |

- **判断できない変更点は `vscode_askQuestions` に含める**。「悩ましい候補」を独り言としてプランに列挙しない
- プランは「これで進める」と了承できる最小限の情報量を目指す
- **`manage_todo_list` の呼び出し頻度**: 6 ステップ以下の直線タスクでは最大 **2 回**（開始時に全ステップ登録・完了時に一括完了）に限定する。中間状態の毎ステップ更新は不要
  - **docs のみタスク（Lean ビルドなし）**: 開始時 1 回登録 + 完了時 1 回更新のみ。中間状態の個別更新は禁止

### ゲーム仕様変更後のコミット前 docs 確認（必須）

ゲーム仕様に関わるコード変更（`S2IL/Shape/Color.lean`, `S2IL/Shape/Shape.lean` 等）を行った場合、コミット前に `docs/shapez2/` の対応する節を確認し、ドキュメントと実装が一致していることを検証すること。ドキュメントに誤りがあれば、コードと一括でコミットする。

### コミット粒度の原則

- **コード変更 + 関連 docs 更新は同一コミット**にまとめる
- docs のみの独立コミットはセッション最終コミットでのみ作成する
- コミットメッセージには sorry 数の変化を含める（例: `sorry: 4→2`）

### ファイル編集ツール選択ルール

以下のケースでは、`replace_string_in_file` / `multi_replace_string_in_file` の `oldString` を構築する前に、必ず同セッション内で `read_file` を実行して実ファイルの正確なテキストを取得すること。

- **10 行以上の連続ブロック**（テーブル・コードブロック・証明ブロック等）を置換する場合
- **前セッションのサマリーや記憶からテキストを引用**しようとする場合
- **`replace_string_in_file` が失敗した直後**に再試行する場合

**Markdown / docs の基本方針**:

- 小規模な見出し修正・表更新・段落差し替えは **`replace_string_in_file` / `multi_replace_string_in_file` を第一選択**とする
- **セクション単位で分割可能な中規模改訂（200〜400 行程度）** は `multi_replace_string_in_file` 複数呼び出しを優先する。VS Code インライン差分が保たれ、変更追跡が容易
- **完全な文書置換（全セクションを一から書き直す等）** に限り **`create_file` + `Move-Item`** を使用する。ただし **`create_file` + `Move-Item` は VS Code エディタのインライン差分バーに反映されない** ため、使用後はユーザーへ「Source Control タブ または `git diff HEAD~` で差分確認」を案内すること
- Markdown 編集後は **必ず `get_changed_files` で差分確認**し、VS Code 上で変更点として追いにくい編集経路を避ける
- docs 単独タスクでは、初回の周辺文書確認後は対象ファイル以外の再読を最小限に抑える。対象が固まった後の同一範囲の反復 `read_file` は避ける

**Lean ファイルの基本方針**:

- 局所的な式差し替え、補題名の更新、短い証明ブロックの差し替えは **`replace_string_in_file` / `multi_replace_string_in_file` を第一選択**とする
- **50 行超の置換**、複数定理にまたがる再構成、長い `match` / `rfl` 連鎖の組み替えは **`create_file` + PowerShell splice** を第一選択とする
- `create_file` + PowerShell splice でファイルを丸ごと置き換えた後は、**`get_changed_files` で差分確認**すること
- Lean ファイルでは繰り返しパターンが多いため、`apply_patch` は原則使わない

**`apply_patch` の使い分け**:

- 正例: 非 Lean ファイルの小規模な単一差分、または 1-2 箇所の局所修正
- 正例: 実ファイルの文脈を `read_file` で確認済みで、差分が短く一意に定まる場合
- 反例: Markdown の章移動・大規模表更新・Appendix 並べ替えなど、レビュー時に差分追跡が重要な編集
- 反例: Lean の `match` 分岐、`rfl` 連鎖、長い証明ブロックの編集
- 反例: `run_in_terminal` から `apply_patch` をシェルコマンドとして直接呼び出すこと。`apply_patch` は専用ツールとして使う

**重要**: `replace_string_in_file` が「multiple matches」や「text not found」で失敗した場合、**ファイルが未修正とは限らない**。失敗理由が「マッチなし」なら、対象箇所がすでに別の値に変更されている可能性がある。失敗時は必ず `grep_search` か `read_file` でファイルの現在の内容を確認してから次の行動を決める。

## コーディング規約

### 基本規約
- 文字コード: UTF-8 (BOMなし)。ただしBOMが必要な場合はBOMを付ける
- 改行コード: CRLF
- コメントは日本語で記述
- インデント: スペース4つ
- 新規ファイルには以下の SPDX ヘッダを追加
  ```lean
  -- SPDX-FileCopyrightText: <作成年> my04337
  -- SPDX-License-Identifier: MIT
  ```
  - 新規作成: `<作成年>` = 現在の年（例: 2026）
  - 大幅改変: 範囲表記に更新（例: 2018-2026）
  - 既存ファイルに SPDX ヘッダが無い場合、追加しない

### Lean ビルド安全習慣

#### API 型シグネチャの事前確認（提案 A）

新規に使用する API の型が不確かな場合、**ビルド前に REPL で `#check` を実行すること**。

特に以下のケースで省略を禁止する:
- Mathlib / Batteries の補題を初めて使う場合
- `Function.*`、`List.*`、`Bool.*` 等の標準ライブラリ関数を引数付きで呼ぶ場合
- `beq_iff_eq` 等の Iff 補題を `(.mp h)` / `(.mpr h)` の形（引数あり）で使おうとする場合

```lean
-- 例: 引数の有無を REPL で確認してからビルド
{"cmd": "#check @beq_iff_eq", "env": 0}
{"cmd": "#check Function.Injective", "env": 0}
```

詳細チェックリスト: [`/memories/repo/repl-quick-checklist.md`](/memories/repo/repl-quick-checklist.md)

#### 多ファイル修正時のビルド戦略（提案 B）

2 ファイル以上を同時に修正する場合、**段階的にビルド検証すること**。
両ファイルのエラーが複合化するとデバッグが困難になるため、以下の手順を守る:

1. **ベースライン確認**: 修正前にビルドが clean であることを確認
2. **依存関係の逆順**: import される側（ライブラリ寄り）のファイルを先に修正・ビルド
3. **単体ビルド**: 各ファイルの変更後、`lake build <Module.Name>` で単独ビルドを確認
4. **統合ビルド**: 全修正完了後、`build.ps1` で全体ビルドを確認

```powershell
# 例: Rotate180Lemmas を先に確認してから Gravity を修正する
lake build S2IL.Behavior.Rotate180Lemmas
# ↑ 成功を確認してから Gravity.lean の修正へ進む
```

## テスト方針

### GameConfig ごとのテストレベル

| レベル | 対象 config | 方針 |
|---|---|---|
| **標準テスト** | `vanilla4`, `vanilla5` | 全操作で網羅的にテスト |
| **ストレステスト** | 8レイヤ等の大規模 config | ボトルネック性質（truncate 冪等性等）のみ |

- テストヘルパーには使用する `GameConfig` を明示する（例: `stackTest` は vanilla4 使用）
- 8レイヤ等のストレステストは `Test/Shape/Shape.lean` に配置し、層ごとの力業 (`cases`) が不可能な規模で性質を検証する
- **`#guard` ベースのテストは `lake build` の `errors = 0` で検証完了**。`#guard` はコンパイル時に評価されるため、ビルド成功 = 全 `#guard` パス。別途 `lake env lean --run` で実行する必要はない

```
pwsh.exe -File .github/skills/lean-build/scripts/build.ps1
bash .github/skills/lean-build/scripts/build.sh
```

- Windows: `.ps1` ファイルは PowerShell で自動的に実行される
- macOS / Linux: `.sh` ファイルは shebang (`#!/usr/bin/env bash`) により適切なシェルで実行される
  - 初回実行時に `chmod +x` が必要な場合がある

### 非同期ビルドの出力取得規則

- `run_in_terminal` を `mode=async` で起動したビルドの監視時は、`get_terminal_output` を**毎ターン連打しない**。同一出力が続く場合は、最低でも 3 回分の別作業（例: 変更確認・diff 確認）を挟んでから再取得する。
- 目安として、同一ビルドに対する `get_terminal_output` の連続呼び出しは **5 回以内**に抑える。5 回を超える場合は監視戦略を切り替える（別タスク先行・最終結果のみ回収）。
